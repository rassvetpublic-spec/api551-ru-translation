# ТЗ API 551 — Stage 4: перевод надписей и сборка 69 составных Figure-объектов

Дата версии: 2026-06-16

Статус: CURRENT / для нового чата в этом же проекте

## 1. Назначение Stage 4

Stage 4 — новый производственный этап после систематизации правил и политик проекта.

Цель Stage 4: получить 69 проверенных составных объектов Figure, которые затем будут использованы в финальной сборке переведённого API 551.

Итог Stage 4 — не финальный PDF всего документа, а approved-набор Figure-объектов:

- approved-рисунки с переведёнными надписями;
- структурные описания составных объектов;
- реестр надписей и решений по каждой надписи;
- проверочные отчёты;
- ZIP-пакеты для приёмки.

## 2. Главный результат

Нужно получить 69 составных Figure-объектов.

Каждый Figure-объект должен быть готов к включению в финальную сборку перевода.

Минимальный состав любого Figure-объекта:

- approved-изображение/чертёж с переведёнными надписями, если надписи должны идти на рисунок;
- название/подпись снизу;
- JSON-запись объекта;
- запись в реестре надписей;
- статус проверки.

Если Figure содержит таблицы, NOTE, длинные пояснения, панели или текстовые блоки, они должны быть оформлены как компоненты составного объекта по approved-правилам.

## 4. Главные правила

1. Не использовать генерацию изображений.
2. Не использовать generative image editing.
3. Работать только программно с существующими PDF/PNG/HTML/JSON/CSV/ZIP.
4. Не дорисовывать и не выдумывать графические элементы.
5. Не удалять линии, стрелки, размеры, оси, рамки, штриховку и другие чертёжные элементы.
6. Очищать только текстовую область.
7. Защищать графику через protected masks.
8. Не использовать старые bbox напрямую.
9. Все сомнительные случаи отправлять в manual-control.
10. В чат выдавать только ZIP/файл + краткая сводка + краткий отчёт проверки.

## 5. Что такое составной Figure-объект

Все рисунки API 551 считаются составными объектами.

Составной Figure-объект может включать:

- основной чертёж/изображение;
- переведённые надписи на изображении;
- подпись/название снизу;
- таблицу расшифровки, если она утверждена как часть объекта;
- реконструированную таблицу;
- NOTE-блоки;
- текстовый абзац над или под рисунком;
- панели/блоки;
- ссылку на исходную PDF-страницу;
- first mention;
- статус approved/manual-control/do-not-change.

В финальном HTML под рисунком должен показываться только approved-рисунок с переведёнными надписями и утверждённые компоненты составного объекта.

Служебные таблицы original/translation, bbox, masks и debug-данные не являются частью чистового вывода, если не утверждены отдельно.

## 6. Реестр надписей

Для каждой надписи нужно определить:

- идёт ли перевод на сам рисунок;
- реконструируется ли блок текстом/таблицей;
- является ли зона protected;
- является ли зона do-not-change;
- требуется ли manual-control;
- какой bbox используется;
- какой mask используется;
- какой approved-перевод принят.

Обязательные решения:

- `on_image` — перевод наносится на рисунок;
- `text_reconstruction` — перевод уходит в текстовый блок;
- `table_reconstruction` — перевод уходит в таблицу;
- `caption_only` — учитывается только в подписи;
- `note_block` — оформляется как NOTE/примечание;
- `protected_no_change` — не менять;
- `manual_control` — ручная проверка.

## 7. BBox и маски

Правила bbox:

- bbox строить максимально близко к буквам;
- bbox = крайние пиксели букв + минимальный отступ;
- не расширять bbox под выносные линии;
- если линия вне надписи, считать её графикой;
- старые bbox использовать только после проверки/нормализации.

Правила масок:

- очистка только по `text_mask`;
- графика защищается через `protected_mask`;
- protected/do-not-change зоны не очищать;
- protected collision zones сохранять.

Координаты:

- `bbox_pdf_points` считать предпочтительным каноническим источником;
- `bbox_crop_pixels` использовать только при подтверждённом crop/asset;
- если bbox сомнителен — manual-control.

## 8. Порядок работы Stage 4

Рекомендуемый цикл по каждому Figure:

1. Найти Figure в approved/reference register.
2. Проверить исходный PDF и asset/crop.
3. Проверить список внутренних надписей.
4. Разнести надписи по решениям: on-image / reconstruction / protected / do-not-change / manual-control.
5. Построить или нормализовать bbox.
6. Сделать bbox-preview.
7. Сделать cleaned-only preview.
8. Проверить, что удалён только текст, а графика сохранена.
9. Нанести русский перевод.
10. Проверить читаемость, шрифт, переносы строк и технический смысл.
11. Сформировать составной Figure-объект.
12. Обновить JSON/CSV-реестр.
13. Сформировать краткий отчёт проверки.
14. Отправить на приёмку.

Для партии или полного набора должны быть:

- краткий текстовый отчёт проверки;
- список manual-control случаев;
- список do-not-change/protected зон;
- список нерешённых вопросов.

## 10. Критерии приёмки Figure-объекта

Figure-объект считается пригодным, если:

- все содержательные надписи обработаны;
- все approved on-image надписи нанесены;
- все text/table reconstruction блоки сохранены;
- NOTE не потеряны;
- подпись/название есть;
- графика не повреждена;
- нет лишнего удаления линий/стрелок/размеров;
- перевод читаем;
- bbox/mask решения трассируемы;
- JSON/CSV обновлены;
- ручные случаи явно перечислены;
- объект готов к финальной сборке.

## 11. Критические ошибки

Figure-объект бракуется, если:

- потеряна содержательная надпись;
- потерян NOTE;
- потеряна таблица внутри Figure;
- повреждена графика;
- удалены линии/стрелки/размеры;
- перевод нанесён в protected/do-not-change зоне;
- bbox взят из старого источника без проверки;
- сомнительный блок заменён автоматически;
- подпись/название отсутствует;
- объект невозможно трассировать по PDF page / section / figure id.
