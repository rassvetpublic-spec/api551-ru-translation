# Техническое задание CURRENT

**Проект:** API 551 — Process Measurement — RU Translation Project  
**Документ-источник:** `API 551 2016 (R2024).pdf`  
**Статус ТЗ:** CURRENT / замена старого `TZ_API551_translation_project_RU.md`  
**Дата фиксации:** 2026-06-16  
**Основание:** актуальная система правил `API551_RULES_*_CURRENT`, `RULES_API551_PDF_FORMATTING_APPROVED`, approved master-реестр надписей, Stage2/Stage3 source packages и текущие решения пользователя.

---

## 1. Цель проекта

Создать технически корректный русский перевод API 551 с сохранением структуры документа, внутренних ссылок, таблиц, рисунков, примечаний, сносок и библиографии.

Итоговые целевые артефакты проекта:

1. **Финальный HTML** — русский чистовик с раскрываемым оригиналом по клику на абзац или объект.
2. **Финальный PDF** — русский печатный чистовик, без английской OCR-колонки и без служебных данных.
3. **JSON/CSV-реестры** — трассировка источника, переводов, рисунков, таблиц, надписей, статусов и QA.
4. **Approved-пакет рисунков** — финальный набор рисунков с утверждёнными русскими надписями и approved-register после принятия всех Figures.

## 2. Главные принципы проекта

### 2.1. Нельзя терять содержательную информацию

Финальные HTML/PDF не должны терять содержательные элементы API 551:

- основной текст;
- заголовки;
- списки;
- таблицы;
- рисунки;
- подписи;
- NOTE/примечания;
- сноски;
- библиографию;
- содержательные внутренние надписи Figure;
- содержательные таблицы и текстовые блоки внутри Figure.

Любое правило «не включать», «не выводить», «не переводить», «исключить из crop» применимо только к служебным слоям: raw OCR, debug, JSON как видимый текст, QA-комментарии, временные anchors, HTML/CSS/JS-интерфейс, внутренние отчёты.

Если фрагмент является исходным содержанием API 551, он должен попасть в итог одним из утверждённых способов: основной русский текст, реконструированная таблица, сноска, примечание, подпись, утверждённый Figure-объект, русская передача внутренних надписей или блок ручной проверки.

### 2.2. Основной текст обрабатывается через OCR страниц

Основной рабочий принцип:

```text
Рендер страницы PDF → OCR → разметка блоков → корректировка OCR → перевод → HTML/JSON → проверка → финальная сборка
```

Встроенный текстовый слой PDF не является основным источником разметки и перевода. Его можно использовать только как вспомогательный источник для сверки спорных мест.

### 2.3. Запрет генеративной обработки изображений

В проекте запрещено:

- использовать image generation;
- генерировать новые изображения;
- применять генеративное редактирование;
- отдавать пользователю сгенерированные картинки вместо программной обработки исходных assets.

Работа с рисунками выполняется только программно с существующими PDF/PNG/HTML/JSON/CSV и approved-реестрами.

### 2.4. Не использовать старые bbox напрямую

Старые bbox нельзя брать как источник истины без проверки. Для замены надписей используются актуальные approved-реестры, оптическая проверка, `text_mask`, `protected_mask`, контроль линий/стрелок/размеров и cleaned-only этап.

## 4. Финальный HTML

Финальный HTML — русский пользовательский чистовик.

Обязательные правила:

1. Видимый основной слой — русский текст.
2. Оригинал скрыт.
3. Оригинал раскрывается по клику на абзац или объект.
4. Клик по ссылке не раскрывает оригинал; ссылка должна работать как ссылка.
5. HTML содержит оглавление, разделы, таблицы, составные объекты рисунков, примечания, сноски, библиографию и внутренние ссылки, если эти элементы входят в сборку.
6. Каждый переводимый элемент хранит трассировку: printed page, PDF page, section, first mention location, ссылка на исходный PDF.
7. Не выводить как видимый чистовой текст raw OCR, debug, bbox/mask overlays, JSON/CSV dumps, TODO, QA-панели.

Под рисунком в финальном HTML показывать только:

- approved-рисунок с переведёнными надписями;
- название/подпись снизу;
- дополнительные текстовые/table-блоки только если они утверждены как часть составного объекта в approved-register.

---

## 5. Финальный PDF

Финальный PDF — русский технический печатный чистовик.

Обязательные правила:

1. Только русский чистовик.
2. Без английской OCR/REVIEW-колонки.
3. Без raw OCR, JSON, HTML/CSS/JS, debug/TODO, внутренних отчётов и служебных комментариев.
4. Потоковая вёрстка по русскому тексту, не копирование исходной пагинации API/PDF.
5. Формат A4, портретная ориентация по умолчанию.
6. Поля: верх 1 cm, низ 1 cm, лево 2 cm, право 1 cm.
7. Основной текст Times New Roman 12 pt.
8. Основные заголовки Times New Roman 14 pt, полужирный.
9. Таблицы 12 pt, допускается 11 pt.
10. Сноски 8 pt.
11. Таблицы реконструировать как таблицы, а не вставлять картинками, если это обычные таблицы документа.
12. Figure и подпись не разделять переносом страницы.
13. Объекты размещать после первого логического упоминания, не переносить в чужой раздел-владелец.
14. Внутренние PDF-ссылки должны вести на правильные якоря итогового PDF.
15. После генерации PDF выполнять проверку по `CHECKLIST_API551_PDF_FORMATTING_APPROVED.md` и `.json`.

Критические условия браковки PDF определяются `RULES_API551_PDF_FORMATTING_APPROVED.md`.

## 6. Рабочие и проверочные HTML

Рабочие HTML используются для OCR, сверки, QA, проверки рисунков, таблиц, bbox, cleaned-only и translated-preview.

Допустимо:

- bilingual page-card;
- OCR/оригинал слева, перевод справа;
- bbox preview;
- cleaned-only preview;
- translated preview;
- side-by-side original/cleaned/translated;
- overlay для контроля bbox;
- списки manual-control, protected zones, do-not-change zones;
- anchors, source ids, bbox ids, QA status, warnings.

Запрещено переносить этот рабочий режим как основной вид финального HTML/PDF.

В чат не выводить HTML-превью, JSON, большие таблицы, PNG-превью и debug dumps. Результаты выдавать файлами/ZIP с краткой сводкой и кратким отчётом проверки.

## 7. Рисунки как составные объекты

Все рисунки API 551 считаются составными объектами.

Минимальный состав любого Figure:

1. approved-рисунок с переведёнными надписями;
2. название/подпись снизу.

Дополнительные компоненты Figure могут включать:

- таблицу расшифровки, если она утверждена как часть объекта;
- NOTE/примечания;
- текстовый блок;
- абзац над или под рисунком;
- панель/блок внутри таблицы;
- реконструированную HTML/PDF-таблицу;
- ссылки на источник и first mention.

Нельзя считать рисунок простым изображением без подписи. Нельзя удалять примечания или текстовые блоки без переноса в утверждённый компонент составного объекта.

## 8. Реестр надписей на рисунках

Реестр надписей определяет:

- какие английские надписи переводятся;
- какой перевод approved;
- что наносится на сам рисунок;
- что реконструируется как текст/таблица;
- что защищается и не меняется;
- что требует ручного контроля.

Основные статусы:

- `approved_on_image` — перевод наносится на сам рисунок;
- `approved_text_reconstruction` — перевод/блок реконструируется как текст или таблица;
- `do_not_change` — не менять;
- `protected` — защищённая зона;
- `manual_control` — требуется ручная проверка;
- `rejected` — отклонено;
- `superseded` — заменено более новой записью.

На сам рисунок идёт только запись со статусом `approved_on_image`.

В текст/таблицу идёт запись со статусом `approved_text_reconstruction`.

Обязательные структуры CSV/JSON заданы в `API551_RULES_05_FIGURE_LABEL_REGISTER_CURRENT.md`.

## 9. Специфические правила по отдельным Figures

Специфические правила хранятся в отдельном живом файле:

- `API551_FIGURE_SPECIFIC_RULES_CURRENT.md`

Текущий зафиксированный статус:

- рисунки приняты до Figure 27 включительно;
- для Figure 27 исправлено NOTE 2;
- следующие приоритетные проходы: Figure 29, Figure 32, Figure 33, Figure 38, Figure 39, Figure 44, Figure 47;
- после принятия всех рисунков сформировать финальный approved-пакет/реестр рисунков;
- финальный approved-пакет/реестр рисунков является обязательным источником для окончательной сборки переведённого документа.

Если правило конкретного Figure конфликтует с общими правилами, конфликт выносится на ручное утверждение. Если правило конкретного Figure конфликтует с approved-register, приоритет имеет последний явно утверждённый approved-register.

## 10. Этапы работы

### 10.1. Подготовка источников

1. Проверить наличие обязательных источников.
2. Проверить наличие актуальных CURRENT/APPROVED правил.
3. Не использовать старые промежуточные источники как approved.
4. Зафиксировать версию реестров и пакетов в отчёте сборки.

### 10.2. Рабочая сборка HTML/JSON

1. Рендер страниц.
2. OCR и разметка блоков.
3. Корректировка OCR.
4. Перевод.
5. Реконструкция обычных таблиц.
6. Вставка Figure как составных объектов.
7. Проверка ссылок и трассировки.
8. Выдача ZIP с HTML/JSON и кратким отчётом.

### 10.3. Работа с рисунками

1. Использовать approved master-реестр надписей.
2. Не использовать старые bbox напрямую.
3. Выполнить cleaned-only этап.
4. Проверить защиту линий, стрелок, размеров, рамок, осей, штриховки.
5. Нанести approved-перевод.
6. Выполнить визуальный и попиксельный контроль.
7. Обновить реестр статусов.
8. После приёмки всех рисунков собрать финальный approved-пакет/реестр рисунков.

### 10.4. Финальная HTML-сборка

1. Русский чистовой слой.
2. Скрытый оригинал по клику.
3. Работающие ссылки.
4. Только approved-рисунки с переведёнными надписями.
5. Без служебных debug/OCR/JSON как видимого текста.

### 10.5. Финальная PDF-сборка

1. Русский печатный чистовик.
2. A4, утверждённые поля и шрифты.
3. Потоковая вёрстка по русскому тексту.
4. Связанные объекты: таблицы, рисунки, подписи, NOTE, сноски.
5. Проверка по approved-чек-листам.
6. Браковка при критических нарушениях.

---

## 11. Правило выдачи пользователю

При выдаче файлов в чат передавать только:

1. ссылку на ZIP/файл;
2. краткую сводку, что сделано;
3. краткий отчёт проверки.

Не выводить в чат HTML, JSON, большие таблицы, картинки/HTML-превью, внутренние отчёты, служебные dumps или длинное содержимое файлов.
