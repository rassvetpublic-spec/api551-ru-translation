Дата: 2026-06-17

Статус: текущий patch-файл к источникам Stage 4. Этот файл дополняет `API551_RULES_SYSTEMATIZED_v5_CLEAN_2026-06-16.zip`, `TZ_API551_STAGE4_FIGURE_OBJECTS_CURRENT_2026-06-16.md`, `api551_approved_label_master_v1.csv` и утвержденные Stage2/Stage3 правила. При конфликте применять более позднее пользовательское утверждение из Stage 4.

Перед любой работой Stage 4 проверять доступность обязательных источников:

Если источник недоступен, остановиться и сообщить: имя файла, роль, проблему доступа и риск для Stage 4.

Запрещено начинать работу по памяти, старым bbox или непроверенным локальным копиям.

## 2. Общие запреты Stage 4

- Не использовать генерацию изображений.
- Не использовать generative image editing.
- Не дорисовывать и не выдумывать графику.
- Работать только программно с существующими PDF/PNG/HTML/JSON/CSV/ZIP.
- Не выводить в чат превью HTML, PNG, большие JSON/CSV/таблицы или внутреннее содержимое отчетов.
- Для пользовательской выдачи давать только ссылку на файл, краткую сводку и краткий отчет проверки.

## 3. Источники истины и приоритеты

Базовый источник утвержденных надписей и переводов для неутвержденных рисунков: `api551_approved_label_master_v1.csv`.

OCR и визуальное распознавание используются только как контроль полноты и размещения. OCR не имеет права сужать список надписей из master CSV и не заменяет утвержденный перевод.

После пользовательского принятия Figure-объекта этот принятый объект становится приоритетом над master CSV для данного рисунка. Если позже пользователь принимает правку рисунка, количество надписей, переводы и решения из принятого/измененного Stage 4 объекта имеют приоритет над CSV.

Поле `action` в CSV не является разрешением игнорировать строку. Каждая строка с `original_text` и/или `ru_text` должна получить явный исход: встроена на PNG, вынесена в таблицу, является caption, protected/no-change, manual-control, fragment/duplicate/split.

## 4. Статусы

Статус должен быть синхронизирован в `catalog.json`, `figure_xxx.object.json`, `object.html` service view и index/catalog.

## 6. HTML-представления

Каждый готовый объект должен иметь два HTML-представления:

- `figure_NNN.out.html` - чистый экспортный объект для финального документа. Не содержит служебной таблицы оригинал/перевод, внутренних bbox, audit markers или service notes.
- `figure_NNN.object.html` - служебный объект для проверки. Показывает финальный PNG, исходный crop PNG, служебную информацию, ссылки на JSON/PNG и таблицу сверки надписей при необходимости.

В `object.html` обязательно показывать:

- финальный PNG;
- оригинальный source crop PNG;
- служебную информацию;
- status;
- ссылку на исходный PDF на нужной странице;
- данные о надписях/защищенных блоках, если они есть.

В `out.html` порядок комплексного объекта:

1. рисунок;
2. таблица надписей, если надписи вынесены из рисунка и должны попасть в итоговый документ;
3. таблица примечаний, если примечания восстановлены из рисунка;
4. название рисунка в самом низу.

Таблицы надписей и примечаний не должны иметь отдельного заголовка таблицы. Для примечаний использовать формат:

```text
Примечание X | Суть
```

## 8. Проверка ZIP перед выдачей

Перед выдачей пользователю проверить:

- ZIP читается, `testzip` без ошибок;
- нет Git LFS pointer вместо реального содержимого;
- PDF не включен в архив;

- статусы в `catalog.json` совпадают с объектными JSON.

## 9. Protected / do-not-translate политика

Не переводить и не трогать на PNG:

- однобуквенные обозначения;
- буквенно-цифровые технические обозначения;
- `NOTE X`, `(NOTE X)` и аналогичные ссылочные маркеры;
- названия и символы химических элементов, когда они являются технической нотацией;
- формулы, переменные, размеры и обозначения типа `H1`, `H2`, `SG 1`, `SG 2`, `SPAN = |URV - LRV|`;
- элементы, явно помеченные `symbol_only`, `dimension`, `protect_do_not_translate`, `protected_no_change`.

Такие элементы можно фиксировать в JSON/service metadata, но исходное изображение на PNG сохраняется.

Если неопределенность выше 0.1%, не гадать и спросить пользователя.

## 10. Полнота надписей и OCR

Для каждого рисунка считать видимые внутренние label-блоки по original PDF crop и сверять с master CSV.

Нужно различать:

- уникальные видимые label-блоки;
- повторы одной и той же надписи;
- protected symbols/dimensions;
- OCR-фрагменты, являющиеся частью многострочного блока;
- объединенные OCR-кандидаты, которые нужно разделить на несколько надписей.

Нельзя классифицировать рисунок как `caption-only`, `note-only` или `без переводимых надписей`, пока original PDF crop визуально/OCR не проверен и CSV не сверена.

Имена assets вроде `notes_removed`, `caption_only`, `accepted`, `stage3_accepted` не доказывают отсутствие переводимых надписей.

## 11. Многострочные надписи

OCR-фрагмент не может становиться самостоятельной надписью, если это часть утвержденного многострочного блока.

Правило начала блока: надпись не должна начинаться с lowercase-фрагмента или середины строки, если только сам утвержденный оригинал явно так не начинается.

Для горизонтальной многострочной надписи искать родительский блок вверх/вблизи.

Для вертикальной многострочной надписи искать родительский блок влево по той же вертикальной колонке.

Если найден fragment-кандидат, использовать bbox родительского блока, а не bbox fragment-кандидата.

## 12. Направление вертикального текста

Вертикальная исходная надпись должна оставаться вертикальной в переводе.

Направление вертикального перевода нельзя определять только по форме bbox. Нужно:

1. Вырезать зону исходной вертикальной надписи из original/source crop.
2. Проверить две нормализации: поворот страницы `+90` и `-90`.
3. Нормализация, в которой оригинальная английская надпись читается правильно, задает направление чтения источника.
4. Русский перевод размещать обратным поворотом на странице так, чтобы при той же нормализации он читался так же, как оригинал.

Пример Figure 37: `Normal working level` читается после clockwise-нормализации страницы, значит русский перевод должен быть counterclockwise на странице.

Для вертикального блока в JSON/service metadata фиксировать: `orientation`, `source_reading_normalization`, `translation_page_rotation`, `direction_policy`, replacement bbox.

## 13. Политика размещения и расширения bbox

При размещении русского перевода:

1. Проверять пиксели слева и справа от окна вставки по всей высоте окна.
2. Если слева и справа свободно, расширять в обе стороны одновременно.
3. Если свободна одна сторона, расширять в свободную сторону.
4. Если слева и справа нет безопасного места, повторить стратегию вверх/вниз и сделать многострочный перевод.
5. Если безопасного места нет, ставить manual-control.

Перед расширением учитывать protected-графику, стрелки, линии, штриховку, размеры и другие do-not-change области.

## 14. Очистка оригинала и фон перевода

Нельзя очищать всю расширенную область поиска/размещения.

Использовать двухпроходную модель:

1. Очистить весь утвержденный bbox оригинальной английской надписи с малым safety margin.
2. Отрисовать русский текст на tight background, рассчитанном по фактическим пикселям текста плюс минимальный padding.
3. Если перевод выходит за оригинальный bbox, расширять только tight background перевода.
4. После отрисовки оптически проверить поля вокруг перевода. Если поля избыточны, перерисовать с уменьшенным background.

Вся оригинальная надпись должна быть удалена, но лишние большие белые поля недопустимы.

## 15. Переносы строк

Если перевод содержит `(...)` или `[...]`, стараться переносить полный скобочный фрагмент на новую строку.

Если длина текста в скобках примерно сопоставима с текстом до скобок, переносить так:

```text
Преобразователь перепада давления
(с вентиляционными клапанами)
```

Внутри скобок делить только если полный скобочный фрагмент не помещается после обычного безопасного расширения.

Сокращение, утвержденное пользователем: `относительная плотность` в переводах сокращать как `отн. плотность`, когда это нужно для размещения.

## 16. Шрифт

## 17. Рисунки с вынесенными надписями и примечаниями

Если встроенные заголовки, note blocks или текстовые блоки исключаются из тела рисунка и выносятся в реконструкцию таблицей:

- на PNG не должно оставаться переводимых английских блоков, которые должны быть реконструированы ниже;
- в итоговом `out.html` после рисунка идет таблица надписей/примечаний без служебных заголовков;
- caption всегда последним элементом;
- service metadata остается только в `object.html`/JSON.

## 20. Acceptance workflow

До принятия пользователем объект имеет review/changed статус и не считается финально approved.

После принятия:

- объект больше не трогать без явного reopening;
- принятие фиксирует количество надписей, переводы и статус;
- следующая overlay-сборка должна включать новый/измененный объект и обновленный catalog/index;
- accepted object становится приоритетом над CSV для данного Figure.
