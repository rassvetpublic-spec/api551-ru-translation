# GITHUB_SKILL_FOR_CHATGPT.md

## Назначение

Навык для работы с GitHub-проектами через ChatGPT/GitHub connector:
- проверка repo, веток, PR, CI и файлов;
- безопасные точечные исправления;
- создание веток и pull request;
- защита от перезаписи неполных файлов;
- разделение source-of-truth, prompts-only и archive/tooling.

## Главное правило

GitHub — источник истины для кода, но не для бизнес-логики проекта, если в проекте есть отдельные ACTIVE/CURRENT источники.

Перед изменением кода всегда выполнить source-gate:
1. определить repo;
2. проверить default branch;
3. проверить права;
4. проверить нужные файлы;
5. определить изменяемую ветку;
6. не менять `master/main` напрямую без явного разрешения;
7. если есть риск неполного fetch — не использовать `update_file` на полный файл.

## Безопасный алгоритм исправления

### 1. Проверить repo

Проверить:
- `repository_full_name`;
- `default_branch`;
- `permissions`;
- `visibility`;
- `archived`.

Если repo archived или нет push/admin прав — остановиться.

### 2. Проверить файл

Для маленького участка:
- читать файл через `fetch_file` с `start_line/end_line`;
- цитировать найденную проблему;
- не полагаться на память.

Для полного файла:
- если `fetch_file` возвращает усечённый результат, не делать полную перезапись файла;
- не использовать неполный контент для `update_file`.

### 3. Для точечного исправления

Предпочтительный путь:
1. создать ветку от default branch;
2. внести минимальное изменение;
3. проверить изменение в новой ветке;
4. открыть PR;
5. дождаться CI;
6. merge только после проверок.

### 4. Если прямой `update_file` опасен

Если файл большой и полный fetch недоступен, можно использовать временный GitHub Actions workflow:

1. создать branch, например `codex-fix-...`;
2. добавить временный workflow в этой branch;
3. workflow сам меняет файл в GitHub runner;
4. workflow удаляет сам себя через `git rm`;
5. workflow делает commit и push в ту же ветку;
6. после выполнения проверить:
   - целевой файл изменён;
   - временный workflow удалён;
   - изменён только нужный файл;
7. открыть PR в default branch.

Этот способ использовать только для простых, проверяемых, точечных правок.

## Запрещено

- не force-push без отдельного разрешения;
- не перезаписывать большой файл неполным fetch;
- не менять `master/main` напрямую, если можно через PR;
- не смешивать tooling repo с прикладным проектом;
- не считать `PROMPTS.md` source-of-truth;
- не удалять файлы без явного подтверждения;
- не скрывать, если действие не удалось;
- не выдавать PR как merged, если он только open.

## Как я исправил `powershell_VBA_sync`

Задача: удалить строку `dfsd sd e3` из `Sync-VBA.ps1`.

Что было сделано:
1. Проверен GitHub repo `rassvetpublic-spec/powershell_VBA_sync`.
2. Проверено, что `master` содержит проблемную строку после `param(...)`.
3. Так как полный `fetch_file` большого `.ps1` был усечён, прямой `update_file` был признан опасным.
4. Создана branch `codex-fix-sync-vba-p0` от `master`.
5. В branch добавлен временный workflow `.github/workflows/auto_fix_sync_vba_p0.yml`.
6. Workflow удалил строку `dfsd sd e3` из `Sync-VBA.ps1`, удалил сам себя и сделал commit.
7. Проверено, что в branch строка исчезла.
8. Создан PR #17 `Fix Sync-VBA parser-breaking test line` в `master`.

Итог:
- `codex-fix-sync-vba-p0` — исправлено;
- `master` — будет исправлен после merge PR;
- изменение точечное: 1 файл, 1 удалённая строка.

## Глобальное подключение навыка

Этот файл можно использовать как постоянный источник правил:
- загрузить в каждый проект, где нужен GitHub;
- вставить краткую версию в Custom Instructions;
- хранить как `GITHUB_SKILL_FOR_CHATGPT.md` в File Library;
- для Codex/GitHub repo можно положить в repo как инструкцию для агента, если используемый инструмент поддерживает repo-level instructions.

Краткая версия для Custom Instructions:

```text
Для GitHub-задач всегда сначала проверяй repo, ветку, права, файлы и CI. Не работай по памяти. Не перезаписывай большой файл, если fetch вернул усечённый контент. Для правок создавай ветку и PR. Master/main напрямую не менять без явного разрешения. PROMPTS.md — только prompts-only. После действия дай repo, branch, PR, изменённые файлы, проверку и следующий шаг.
```
