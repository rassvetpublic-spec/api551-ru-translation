Status: CURRENT candidate, 2026-06-18.

Stage 4 must produce 69 approved composite Figure objects for final document assembly.

Every figure is a composite object. A figure is never just a loose image.

Minimum composite object:

- final/review PNG;
- figure caption at the bottom.

Possible components:

- drawing/image;
- caption;
- label table;
- note table;
- restored text block;
- reconstructed HTML/PDF table;
- source PDF link;
- service metadata in JSON and review HTML only.

## 2. Final Object Order

For every clean export object:

1. figure image;
2. label table, if labels are intentionally moved out of the image;
3. notes table, if notes are restored from the figure;
4. figure caption at the bottom.

Caption must be the last visible component.

Do not add a separate visible table title for label or note tables.

Note row format:

`Примечание X | text`

- `figure_NNN.out.html` - clean export object for final document assembly.

- service audit tables;
- bbox coordinates;
- internal comments;
- `Служебная`;
- `Оригинал`;
- `Источник bbox`;

`object.html` must show:

- final PNG;
- original source crop PNG;
- status;
- source PDF link;
- links to local object files;

# API 551 Stage 4 - 02 Labels, OCR, Translation, and Placement

Status: CURRENT candidate, 2026-06-18.

Purpose: consolidated policy for figure labels, OCR verification, protected text, translation placement, clearing, wrapping, vertical text, and figure-specific correction lessons.

## 1. Label Source

For unaccepted figures, `api551_approved_label_master_v1.csv` is the baseline label and translation source.

For accepted figures, the accepted Stage 4 object overrides CSV for that figure.

The CSV `action` field is not permission to ignore a row. Every row with `original_text` or `ru_text` must receive an explicit outcome:

- on-image replacement;
- text reconstruction;
- table reconstruction;
- caption;
- protected/no-change;
- manual-control;
- fragment;
- duplicate;
- split.

## 2. OCR Role

OCR is verification and discovery support only.

OCR must be used to:

- check visible labels on the original PDF crop;
- check labels on Stage2/Stage3 production crops;
- detect missed labels;
- detect fragments and multiline grouping errors;
- detect split/merged candidates.

OCR must not:

- override approved translations;
- reduce the CSV label list;
- classify a figure as label-free without CSV and visual verification;
- rely on asset names such as `notes_removed`, `caption_only`, `accepted`, or `stage3_accepted`.

## 3. Protected / Do-Not-Translate Policy

Do not translate or alter on the PNG:

- one-letter symbols;
- one-digit symbols;
- alphanumeric technical tags;
- `NOTE X`, `(NOTE X)`, `Note X` references;
- chemical element names/symbols when used as technical notation;
- formulas;
- variables;
- dimensions and values with units;
- section markers such as `Section A-A`;
- service abbreviations;
- symbol-only labels;
- protected collision zones;
- items explicitly marked `symbol_only`, `dimension`, `protect_do_not_translate`, or `protected_no_change`.

Protected items may be recorded in JSON/service metadata, but the source image remains visually unchanged.

## 4. Completeness Check

Before deciding that a figure has no translatable labels:

1. Read CSV rows for the figure.
2. Render or load the original PDF crop.
3. Run OCR/text-layer/visual check on the original crop.
4. Run OCR/visual check on the production crop.
5. Compare detected visible label blocks against CSV rows.
6. Document every row outcome.

If uncertainty is greater than 0.1%, ask the user.

## 5. Multiline Labels

Group OCR fragments into semantic source labels before translation placement.

A label normally cannot start with a lowercase fragment or a middle fragment unless the approved source text itself starts that way.

For horizontal multiline labels, search upward and nearby for the parent block.

For vertical multiline labels, search leftward along the same vertical column for the parent block.

If a fragment belongs to a parent label, use the parent label bbox, not the fragment bbox.

## 6. Vertical Text Direction

Do not infer vertical text direction from bbox shape alone.

For every vertical source label:

1. Crop the original label zone from the original/source crop.
2. Test two normalizations: page rotation `+90` and `-90`.
3. The readable normalization defines source reading direction.
4. Place Russian text with inverse page rotation so it is readable under the same normalization.

Apply multiline grouping before determining vertical direction.

Record in JSON/service metadata:

- `orientation`;
- `source_reading_normalization`;
- `translation_page_rotation`;
- `direction_policy`;
- replacement bbox.

Figure 37 correction:

- `Normal working level` is readable after clockwise page rotation.
- Russian translation must therefore be counterclockwise on the page.

## 7. Placement Expansion

When placing Russian text:

1. Inspect pixels left and right of the placement window across the full window height.
2. If both sides are clear, expand both directions.
3. If only one side is clear, expand to that side.
4. If neither side is clear, repeat the strategy vertically and use multiline text.
5. If no safe space exists, mark manual-control.

Before expansion, respect protected graphics:

- leader lines;
- arrows;
- drawing lines;
- hatching;
- table borders;
- dimensions;
- masks;
- do-not-change zones.

## 8. Clearing and Background

Do not clear the full expanded placement/search rectangle.

Use two-pass clearing:

1. Clear the full approved source label bbox with a small safety margin.
2. Draw Russian text on a tight background calculated from actual rendered text pixels plus minimal padding.
3. If translation extends beyond source bbox, expand only the tight translation background.
4. Optically check margins after drawing and redraw tighter if background is excessive.

The full original label must be removed. Large white fields around translation are defects.

## 9. Wrapping

If translation contains `(...)` or `[...]`, try to move the full bracketed part to a new line.

If the bracketed part is roughly comparable in length to preceding text, prefer:

`Преобразователь перепада давления`

`(с вентиляционными клапанами)`

Split inside brackets only if the whole bracketed fragment still cannot fit after safe expansion.

Use `отн. плотность` instead of `относительная плотность` where space-constrained.

## 10. Font

Fixed programmatic font for PNG label replacement:

`/usr/share/fonts/opentype/urw-base35/NimbusSans-Regular.otf`

If unavailable, stop and report. Do not silently substitute another font.

## 11. Figure-Specific Corrections

Figure 15:

- Do not miss `Total vacuum (zero or absolute)`, `Absolute`, `Barometric range`, and atmospheric pressure text.

Figure 37:

- Vertical fragments such as `level` are not independent when part of `Normal working level`.
- Direction must be determined by crop rotation test.

Figure 46:

- `Area to avoid` and `Preferred tap locations` must be accounted for.
- Suspected untranslated text must first be checked as possible part of a multiline already translated label.

Figure 47:

- Mixed object, not note-only.
- `notes_removed` does not mean no callouts.
- Required callouts include `From process top`, `Max liquid level`, and `To instrument`.
- Note references are protected/manual-control, not ordinary labels.

Figure 48:

- Use parent-block bbox, not fragment bbox.
- Tight clearing is mandatory.

Figure 51:

- Changed/review only.
- Rebuild from clean source crop, not old translated output.

Figure 62:

- Visually check split combined OCR candidates.
- One OCR candidate may represent two separate labels.

Figure 66:

- Not title-only PNG.

# RULES_API551_STAGE2_IMAGE_LABEL_REPLACEMENT_STRATEGIES_v1

**Проект:** API 551 — Process Measurement — RU Translation Project  
**Назначение:** стратегии программной замены английских надписей внутри рисунков на русские переводы  
**Статус:** USER-APPROVED STRATEGY DRAFT / для загрузки в источники проекта  
**Дата фиксации:** 2026-06-10  
**База принятой стратегии:** `fig03_03_stagec_panel_ru_reconstructed.png`, пакеты `STAGE2 FULL v2` и последующие пользовательские правки.

---

## 0. Главный принцип

Замена надписей внутри рисунков API 551 выполняется только программной обработкой существующих изображений и реестров. Генеративное создание или генеративное редактирование изображений в этом проекте запрещено.

Принятая рабочая модель:

```text
исходный PNG/PDF → точное определение области английской надписи → локальная очистка → нанесение русского перевода → попиксельный контроль → HTML/JSON/CSV-аудит
```

---

## S01 — запрет генеративной обработки

В этом чате и в проекте API 551:

- не использовать `image generation`;
- не генерировать изображения в чате;
- не применять генеративное редактирование;
- работать только программно с существующими `PNG / PDF / HTML / JSON / CSV`.

Если формулировка запроса звучит как «создай изображение», «сделай картинку», «перерисуй», «сгенерируй PNG», исполнитель обязан сначала уточнить, требуется ли именно генерация. По умолчанию выполнять только программную обработку существующих assets.

---

## S02 — базовая стратегия замены надписей

Основная стратегия для всех изображений:

1. найти исходную английскую надпись;
2. определить её точный bbox;
3. локально очистить исходный английский текст;
4. сохранить линии, стрелки, размеры, контуры, штриховку и таблицы;
5. нанести русский перевод в очищенную область;
6. при необходимости безопасно расширить поле;
7. выполнить визуальный и попиксельный контроль.

Базовый эталон стратегии — успешная обработка `fig03_03_stagec_panel_ru_reconstructed.png`.

---

## S03 — стратегия точной очистки без прямого наложения

Прямое наложение русского перевода поверх английской надписи **не использовать**.

Правильный порядок:

1. сначала найти точную область исходной английской надписи;
2. очистить английскую надпись локально, минимальной зоной;
3. очистка должна удалять только буквы/текстовую область;
4. очистка не должна задевать:
   - выносные линии;
   - стрелки;
   - размерные линии;
   - контуры чертежа;
   - штриховку;
   - таблицы;
   - соседние надписи;
5. после очистки нанести русский перевод в очищенную область или безопасно расширенную область;
6. если перевод не помещается — сначала работать переносами, положением и шириной поля, а не уменьшением шрифта;
7. после нанесения выполнить визуальный и попиксельный контроль.

Короткая формула:

```text
очистка → контроль чистой зоны → нанесение перевода → контроль наложений
```

---

## S04 — запрет множественных перекрытий

Перевод не должен:

- накладываться на другие надписи;
- перекрывать линии;
- перекрывать стрелки;
- перекрывать размеры;
- перекрывать таблицы;
- залезать на соседние графические элементы.

Если найдено множественное перекрытие — рисунок не принимать, переделывать.

---

## S05 — единый размер шрифта внутри одного рисунка

Внутри одного рисунка стараться держать переведённые надписи **одинакового размера**.

Допускается небольшое отличие только если без этого надпись не помещается.

Приоритет:

1. читаемость;
2. единообразие;
3. отсутствие перекрытий;
4. сохранение технической точности.

---

## S06 — не уменьшать шрифт без необходимости

Если перевод не помещается:

1. изменить переносы строк;
2. расширить поле в безопасную сторону;
3. сдвинуть текст внутри зоны;
4. использовать многострочную компоновку;
5. только потом уменьшать шрифт.

Запрещено делать шрифт мелким, если проблему можно решить компоновкой.

---

## S07 — стратегия расширения поля

Направление расширения зависит от положения надписи:

- надпись левее чертежа — расширять поле влево;
- надпись правее чертежа — расширять поле вправо;
- надпись внутри чертежа — расширять пропорционально и осторожно;
- рядом с выноской — не закрывать выносную линию;
- рядом с размером — не закрывать размерную линию и стрелки.

---

## S08 — многострочные надписи

Многострочная надпись считается одним смысловым блоком.

В реестре хранить:

- один `original_text`;
- один `ru_text`;
- один общий `bbox` всей многострочной надписи.

Не хранить координаты каждой строки как самостоятельные независимые надписи, если это один смысловой блок.

---

## S09 — строки с маленькой буквы

Если OCR-строка начинается с маленькой буквы, то почти всегда это продолжение многострочной надписи.

Действие:

1. искать выше начало блока;
2. проверять смысловую связь;
3. проверять геометрию выноски;
4. объединять строки в одну надпись;
5. не переводить нижнюю строку отдельно.

Пример:

```text
connection is 0.07 x Vessel OD
```

Это продолжение блока:

```text
Minimum hillside connection is 0.07 x Vessel OD
```

---

## S10 — группировка по выноске

Строки объединяются не только по близости, а по смыслу и геометрии выноски.

Если к строкам подходит одна выносная линия — это один блок.

Если рядом две строки, но к ним идут разные выноски — это разные блоки.

---

## S11 — сохранение направления надписи

Направление русского перевода должно соответствовать направлению оригинала:

- горизонтальная надпись → горизонтально;
- вертикальная надпись → вертикально;
- наклонная надпись → ручной контроль.

Пример: `Normal working level` на Figure 37 — вертикальная надпись.

---

## S12 — непереводимые элементы

Не переводить, но учитывать как защищённые зоны:

- одиночные буквы и цифры;
- линейные размеры;
- диаметры;
- углы типа `120°`;
- размеры типа `1 1/2 in.`;
- `Section A-A`;
- цифры на осях графиков;
- кандидаты шрифтом менее 5 pt;
- `(Note X)`;
- `MIN`, `MAX`, `TYP`;
- `NPS`;
- `Flange`, если это не самостоятельная содержательная подпись, согласованная отдельно;
- химические формулы и среды типа `C2H5`, `Air/N2`.

Эти элементы не переводятся, но должны учитываться, чтобы русский текст на них не накладывался.

---

## S13 — NOTE-блоки внутри рисунков

Если NOTE-блок уже вынесен в текстовую реконструкцию/перевод ниже рисунка, не заменять его внутри PNG.

Особенно важно для Figure 47 и подобных рисунков.

---

## S14 — не захватывать текст ниже рисунка

Не подхватывать подписи, абзацы и пояснения ниже графической области, если они уже оформлены как текстовые поля.

Пример: Figure 38 — длинный текст про смещение штока не должен повторно внедряться в PNG, если он уже вынесен в текстовую реконструкцию.

---

## S15 — защита принятых рисунков

Если рисунок принят, он больше не меняется, пока пользователь явно не попросит.

Текущий статус:

- пакет `STAGE2 FULL v2` принят как база;
- дальше обрабатываются только недостающие или явно указанные проблемные рисунки.

---

## S16 — контроль после каждого рисунка

После каждого подготовленного рисунка проверять:

- русский текст читается;
- английский текст удалён только в нужных местах;
- нет наложений;
- линии и стрелки не повреждены;
- размеры изображения сохранены;
- изменения вне разрешённых зон отсутствуют или объяснены.

---

## S17 — попиксельный контроль

Для каждого изменения фиксировать:

- разрешённые зоны редактирования;
- количество изменённых пикселей внутри зон;
- количество изменённых пикселей вне зон.

Если вне разрешённых зон изменений больше нуля — это риск и нужен анализ.

---

## S18 — контроль crop

Перед заменой проверять crop:

- верхняя граница не должна резать буквы;
- нижняя граница не должна резать подписи, если они входят в рисунок;
- боковые границы не должны резать надписи или выноски;
- если crop режет текст — сначала корректировать crop, потом заменять надписи.

---

## S19 — оформление HTML-проверки

В HTML-проверке:

- показывать оригинал рядом с реконструкцией;
- не показывать bbox-рамки по умолчанию;
- после реконструкции добавлять свернутое поле **«Заменённые надписи»**;
- прямую ссылку на HTML в чат не выдавать;
- HTML только внутри ZIP.

---

## S20 — вывод результата в чат

После редактирования файлов в чат выдавать только:

1. ссылку на ZIP или созданный файл;
2. краткую сводку;
3. краткий отчёт проверки.

Не выводить:

- HTML-превью;
- прямую ссылку на HTML;
- изображения;
- JSON;
- CSV;
- большие таблицы;
- debug-данные.

---

## S21 — кандидаты на перевод

В кандидаты включать только осмысленные английские слова/фразы.

Не включать:

- технический шум;
- размеры;
- символы;
- оси графиков;
- служебные сокращения;
- элементы, уже вынесенные в текстовую реконструкцию.

---

## S22 — координатная стратегия

Для точной привязки:

1. искать надпись в оригинальном API 551 PDF;
2. определять привязку к текущему PNG через якорь, диагональ и масштаб;
3. пересчитывать координаты в текущий asset;
4. уточнять bbox по пикселям;
5. для многострочного блока сохранять общий прямоугольник.

---

## S23 — стратегия при наличии перевода в источниках

Если строка уже содержится в источниках перевода:

- искать ранее распознанный английский `original_text`;
- по нему определять точные границы;
- использовать перевод из утверждённых данных.

Если текста нет в источниках:

- перепроверять место с большей точностью;
- делать кандидата;
- не заменять автоматически без уверенности.

---

## S24 — контроль ложных повторов

Не переводить одну и ту же надпись дважды.

Проверять:

- не является ли строка частью многострочного блока;
- не входит ли она в заголовок таблицы;
- не повторяется ли текст в нижнем текстовом поле;
- не захвачен ли текст вне рисунка.

Примеры:

- Figure 12: `alignment assembly` — часть `Ball joint lens alignment assembly`;
- Figure 34: `Nozzle exclusion zone...` — только одно вхождение;

- не менять уже принятые рисунки;
- выдавать отдельный пакет только с недостающими;

# API 551 — правила финального HTML v4

Дата версии: 2026-06-16

## 1. Назначение

Финальный HTML — чистовой пользовательский документ на русском языке. Он предназначен для чтения, проверки структуры и навигации, а не для показа служебных OCR/debug-данных.

## 2. Основной принцип

- Видимый основной слой HTML — русский чистовик.
- Оригинал скрыт.
- Оригинал раскрывается по клику на абзац или объект.
- Клик по ссылке не раскрывает оригинал; ссылка должна работать как ссылка.
- Раскрываемый оригинал нужен для проверки и трассируемости, но не должен визуально перегружать чистовик.

## 3. Структура HTML

Финальный HTML должен содержать:

- оглавление;
- разделы и подразделы;
- русский основной текст;
- таблицы;
- составные объекты рисунков;
- примечания;
- сноски или ссылки на сноски, если они есть в исходнике;
- внутренние ссылки на разделы, таблицы и рисунки;
- ссылки на страницы оригинального PDF.

Каждый переводимый элемент должен хранить трассировку:

- original printed page;
- PDF page;
- section;
- first mention location;
- ссылка на исходный PDF вида `API 551 2016 (R2024).pdf#page=N`.

## 4. Раскрываемый оригинал

Оригинал раскрывается для:

- абзаца;
- заголовочного блока, если это нужно для проверки;
- таблицы;
- рисунка как составного объекта;
- примечания;
- сноски;
- библиографической записи.

Оригинал не раскрывается при клике на:

- внутреннюю ссылку;
- ссылку на PDF;
- ссылку на таблицу;
- ссылку на рисунок;
- элемент управления интерфейсом.

## 5. Рисунки в финальном HTML

Все рисунки являются составными объектами.

В финальном HTML под рисунком показывать:

- только approved-рисунок с переведёнными надписями;
- название/подпись снизу как часть составного объекта.

Не показывать под финальным рисунком автоматически:

- таблицу расшифровки всех надписей;
- original/translation пары;
- служебный реестр;
- bbox;
- mask;
- debug-данные.

Исключение: если approved-register явно требует текстовый реконструированный блок как часть составного объекта, он включается по правилам `API551_RULES_04_FIGURE_COMPOSITE_OBJECTS_v4_2026-06-16.md`.

## 6. Таблицы в финальном HTML

Таблицы должны быть реконструированы как HTML-таблицы.

Требования:

- сохранить объединения строк и столбцов;
- сохранить технически значимое выравнивание;
- сохранить жирность через `<strong>`;
- перевести примечания;
- не терять Notes и многострочные пояснения;
- обеспечить перенос текста внутри ячеек;
- не выводить raw OCR как видимый текст.

## 7. Ссылки

В финальном HTML должны работать:

- ссылки на разделы;
- ссылки на таблицы;
- ссылки на рисунки;
- ссылки на примечания;
- ссылки на страницы исходного PDF.

Если целевой объект есть в HTML, ссылка ведёт на якорь HTML.

Если целевого объекта нет в HTML, ссылка ведёт на страницу исходного PDF.

## 8. Интерфейс финального HTML

Допускаются:

- переключение размера шрифта;
- режим компактного/обычного чтения;
- раскрытие/сворачивание оригинала;
- поиск по документу средствами браузера.

Не допускается:

- видимые debug-панели;
- видимые bbox/mask overlays;
- видимые JSON/CSV dumps;
- служебные TODO;
- сырые OCR-блоки как часть чистовика.

## 9. Отличие от рабочих HTML

Финальный HTML не является QA-превью.

В финальном HTML не показываются:

- OCR слева / перевод справа как основной режим;
- bbox preview;
- cleaned-only preview;
- translated preview;
- списки ошибок;
- внутренние технические таблицы.

Эти данные относятся к рабочим и проверочным HTML.

# API 551 — правила рабочих и проверочных HTML v4

Дата версии: 2026-06-16

## 1. Назначение

Рабочие и проверочные HTML используются для OCR, сверки, QA, проверки рисунков, таблиц, bbox, cleaned-only и translated-preview.

Они не являются финальным чистовиком.

## 2. Рабочий bilingual-режим

Допустим рабочий режим:

- одна PDF/API-страница = один `page-card`;
- внутри смыслового блока оригинал/OCR и перевод идут парой;
- оригинал/OCR слева;
- русский перевод справа.

Этот режим нужен для проверки и не должен переноситься как основной вид финального PDF.

## 3. Рабочий HTML Stage 1

Правила:

- основной текст брать через OCR изображений страниц;
- embedded PDF text layer не использовать как основной источник текста;
- страница разбирается на текстовые блоки, таблицы и placeholders рисунков;
- внутренние надписи на рисунках в Stage 1 не переводить прямо на изображении;
- внутренний анализ рисунков хранить в JSON;
- обычные таблицы реконструировать как HTML-таблицы;
- внутренние ссылки сохранять рабочими.

## 4. Проверочные HTML для рисунков

Допустимые проверочные виды:

- bbox preview;
- cleaned-only preview;
- translated preview;
- side-by-side original/cleaned/translated;
- overlay для контроля bbox;
- список manual-control блоков;
- список protected zones;
- список do-not-change zones.

Эти виды не выводятся в чат как HTML-превью. Они передаются только файлами/ZIP.

## 5. BBox preview

Правила:

- bbox строить максимально близко к буквам;
- bbox = крайние пиксели букв + минимальный отступ 1 px;
- допускается контрольная рамка 1 px;
- выносные линии не включать в bbox;
- если линия вне надписи, считать её графикой;
- старые bbox без пересчёта не использовать.

## 6. Cleaned-only preview

Правила:

- сначала формировать cleaned-only;
- удалять исходный текст полностью, включая anti-alias outline;
- очищать только по `text_mask`;
- не удалять protected dark pixels;
- линии, оси, рамки, стрелки, leader-lines, размеры и штриховки защищать.

## 7. Translated preview

Правила:

- наносить русский перевод только после проверки cleaned-only;
- сохранять читаемость;
- не нарушать чертёж;
- многострочный блок держать как один смысловой блок;
- шрифт не меньше 8 pt;
- размеры шрифта внутри одного рисунка не должны отличаться более чем на 2 pt без причины.

## 8. Служебные данные

В рабочих HTML допустимы:

- anchors;
- source ids;
- bbox ids;
- QA status;
- warnings;
- manual-control marks;
- links to source PDF page;
- internal audit links.

Но они должны быть явно отделены от чистового слоя и не попадать в финальный HTML/PDF как видимый текст.

## 9. Интерфейс рабочих HTML

Допускаются:

- управление размером шрифта;
- управление вертикальным отступом ячеек;
- управление горизонтальным отступом ячеек;
- раскрытие технических деталей;
- фильтры по статусам;
- переходы по anchors.

## 10. Выдача пользователю

В чат не вставлять:

- HTML-превью;
- JSON;
- большие таблицы;
- PNG-превью;
- debug dumps.

Выдавать:

- ZIP;
- краткую сводку;
- краткий отчёт проверки.

# API 551 — правила рисунков как составных объектов v4

Дата версии: 2026-06-16

## 1. Основной принцип

Все рисунки API 551 считаются составными объектами.

Минимальный состав любого рисунка:

- approved-рисунок с переведёнными надписями;
- название/подпись снизу.

## 2. Возможные компоненты составного рисунка

Составной объект Figure может включать:

- изображение/чертёж;
- название/подпись;
- таблицу расшифровки;
- примечания;
- текстовый блок;
- абзац над рисунком;
- абзац под рисунком;
- панель/блок внутри таблицы;
- реконструированную HTML/PDF-таблицу;
- ссылки на источник и first mention.

## 3. Финальный HTML

В финальном HTML под рисунком показывать только:

- approved-рисунок с переведёнными надписями;
- название/подпись снизу;
- дополнительные текстовые/table-блоки только если они утверждены как часть составного объекта в approved-register.

Не показывать служебную таблицу original label / translation, если она не утверждена как часть составного объекта.

## 4. Финальный PDF

В финальном PDF рисунок отображается как русский чистовой составной объект.

Правила:

- использовать approved-рисунок с переведёнными надписями;
- название/подпись снизу;
- не отделять подпись от рисунка переносом страницы;
- не выводить служебные расшифровки, если они не являются частью утверждённого объекта.

## 5. Когда надпись идёт на сам рисунок

Надпись идёт на сам рисунок, если:

- это осознанная английская техническая фраза;
- она является внутренней подписью/лейблом чертежа;
- есть approved-перевод;
- есть допустимый bbox;
- перевод помещается без нарушения графики;
- зона не protected/do-not-change;
- объект не утверждён как текстовая реконструкция.

## 6. Когда надпись реконструируется текстом/таблицей

Надпись или блок реконструируется в текст/таблицу, если:

- это длинный поясняющий текст;
- это NOTE-блок;
- это таблица внутри Figure;
- это многострочное пояснение, которое технически лучше читать как текст;
- перевод не помещается в исходную область без разрушения чертежа;
- блок уже утверждён как table/text reconstruction;
- текущий Figure-specific rule требует таблицу, абзац или блок ниже/выше рисунка.

## 7. Оригинал и трассировка

Для каждого составного объекта хранить:

- original printed page;
- PDF page;
- section;
- first mention location;
- source figure id;
- asset id/path;
- approved status;
- caption original;
- caption RU;
- component list;
- links to source PDF page.

## 8. Запреты

Запрещено:

- считать рисунок простым изображением без подписи;
- терять название/подпись;
- удалять примечания без переноса в утверждённый текстовый блок;
- смешивать служебный реестр надписей с чистовым составным объектом;
- подменять approved-рисунок оригинальным рисунком без перевода после принятия approved-пакета;
- автоматически решать спор “на рисунок или в таблицу” без approved-статуса.

## 9. Связь с Figure-specific rules

Конкретные исключения по Figure хранятся в `API551_FIGURE_SPECIFIC_RULES_CURRENT_v2_2026-06-16.md`.

Если правило конкретного рисунка конфликтует с этим документом, требуется ручное утверждение.

# API 551 — правила реестра надписей на рисунках v4

Дата версии: 2026-06-16

## 1. Назначение

Реестр надписей управляет тем:

- какие английские надписи переводятся;
- какой перевод approved;
- что наносится на сам рисунок;
- что реконструируется как текст/таблица;
- что защищается и не меняется;
- что требует ручного контроля.

Главный approved master-реестр: `api551_approved_label_master_v1.csv`, если не заменён более новым approved-файлом.

## 2. Статусы

Обязательные статусы:

- `approved_on_image` — перевод наносится на сам рисунок;
- `approved_text_reconstruction` — перевод/блок реконструируется как текст или таблица;
- `do_not_change` — не менять;
- `protected` — защищённая зона;
- `manual_control` — требуется ручная проверка;
- `rejected` — отклонено;
- `superseded` — заменено более новой записью.

## 3. Что идёт на сам рисунок

На сам рисунок идёт только запись со статусом `approved_on_image`.

Условия:

- есть original label;
- есть approved Russian translation;
- есть bbox или способ получить bbox;
- зона не protected;
- зона не do-not-change;
- перевод технически помещается;
- стиль соответствует правилам надписей;
- замена не разрушает графику.

## 4. Что реконструируется текстом/таблицей

В текст/таблицу идёт запись со статусом `approved_text_reconstruction`.

Типовые случаи:

- длинные абзацы;
- NOTE-блоки;
- таблицы внутри рисунка;
- пояснительные панели;
- многострочные блоки, которые лучше читать как текст;
- блоки, которые нельзя безопасно заменить внутри PNG/PDF-asset.

Если текст перенесён в HTML/PDF-реконструкцию, его не заменять внутри PNG.

## 5. Что не переводится

Не включать в кандидаты:

- одиночные буквы;
- одиночные цифры;
- размеры;
- диаметры;
- углы;
- значения с единицами;
- `Section A-A` и аналогичные сечения;
- цифры на осях;
- кандидаты со шрифтом меньше 5 pt;
- `(Note X)`;
- `MIN`;
- `MAX`;
- `TYP`;
- `NPS`;
- `NPT`;
- химические формулы;
- зоны `protected_collision_zone`.

# RULES_API551_PDF_FORMATTING_APPROVED

Проект: API 551 — Process Measurement — RU Translation Project

Назначение: утверждённые правила финальной печатной PDF-вёрстки на русском языке.

Статус: APPROVED / v5-patch по `RULES_API551_PDF_FORMATTING_VERIFIED_v2`

Дата фиксации: 2026-06-16

Основание: систематизация v4 + полный перенос правил из `RULES_API551_PDF_FORMATTING_VERIFIED_v2`, предоставленного пользователем в чате 2026-06-16.

## 0. Главный запрет на потерю информации

Финальный русский PDF не должен терять содержательную информацию источника.

Любое правило вида «не включать», «не выводить», «не переводить», «исключить из crop», «не вставлять в PDF» разрешено применять только к служебным слоям и техническим артефактам сборки:

- raw OCR;
- английской REVIEW-колонке;
- JSON;
- HTML/CSS/JS интерфейсу;
- отчётам проверки;
- внутренним техническим комментариям;
- debug/TODO/служебным маркерам;
- дублирующим служебным данным.

Если фрагмент является исходным содержанием API 551, он должен попасть в итоговый файл одним из допустимых способов:

1. как переведённый основной текст;
2. как реконструированная таблица;
3. как сноска;
4. как примечание;
5. как подпись рисунка;
6. как оригинальный Figure-crop на рабочих стадиях;
7. как русская передача внутренних надписей Figure;
8. как отдельный блок ручной проверки, если автоматическая классификация сомнительна.

Нельзя использовать формулировки «этот абзац не включать», если абзац является основным текстом источника. Правильная формулировка: «абзац относится к основному тексту, переводится в основном потоке; crop рисунка начинается после него».

## 1. Назначение финального PDF

Финальный PDF — русская техническая печатная версия API 551.

Он должен содержать, если соответствующие элементы входят в финальную сборку:

- титульные/служебные страницы;
- основной русский текст;
- нумерованные разделы и подразделы;
- списки всех уровней;
- таблицы;
- рисунки;
- подписи рисунков;
- примечания;
- сноски;
- содержание;
- перечни рисунков/таблиц;
- библиографию/список ссылочных документов;
- рабочие внутренние PDF-ссылки.

Финальный PDF не является двухколоночным REVIEW-документом и не должен содержать английскую OCR-колонку.

## 2. Базовый принцип вёрстки

Итоговая PDF-вёрстка выполняется по русскому переведённому тексту непрерывным потоком.

Запрещено жёстко повторять постраничную структуру исходного английского PDF по принципу «одна страница источника = одна страница перевода».

Правильная логика:

1. русский текст идёт сверху вниз в пределах доступной области страницы;
2. после заполнения области страница завершается;
3. следующий блок переходит на следующую страницу;
4. перенос выполняется по русскому тексту, а не по исходной английской странице;
5. таблицы, рисунки, подписи, примечания и сноски размещаются как связанные объекты.

## 3. Формат страницы и поля

- Формат: A4.
- Ориентация: портретная по умолчанию.
- Верхнее поле: 1 cm.
- Нижнее поле: 1 cm.
- Левое поле: 2 cm.
- Правое поле: 1 cm.

Дополнительные правила:

1. Печатная область считается после вычета полей и зон колонтитулов.
2. Верхний колонтитул не должен перекрываться основным текстом.
3. Нижний колонтитул не должен перекрываться текстом, таблицами, рисунками, сносками или подписями.
4. Ширина любого объекта не должна выходить за правую границу печатной области.
5. Альбомная страница не применяется автоматически.
6. Если таблица или Figure не помещается в портретную A4-область даже после допустимого уменьшения, объект выносится в список ручной проверки.

## 4. Колонтитулы и нумерация страниц

1. Колонтитулы формируются до размещения основного текста.
2. Колонтитулы не должны визуально доминировать над основным текстом.
3. Номер страницы итогового PDF ставится по русскому потоку, а не по номеру страницы исходного API/PDF.
4. При необходимости в колонтитуле или служебной зоне можно сохранять краткую привязку к исходному документу, но она не должна мешать чтению.
5. Номера исходных PDF/API-страниц используются в JSON/отчётах и ссылках проверки; в финальном печатном тексте они не должны создавать впечатление основной нумерации русского PDF.

## 5. Шрифты и размеры

- Основной текст: Times New Roman, 12 pt.
- Основные нумерованные заголовки: Times New Roman, 14 pt, полужирный.
- Вложенные технические заголовки: Times New Roman, 12-14 pt, полужирный; применять единообразно по всему PDF.
- Текст таблиц: Times New Roman, 12 pt; допускается 11 pt.
- Подписи таблиц: Times New Roman, 12 pt, полужирный или обычный по единому стилю.
- Подписи рисунков: Times New Roman, 12 pt, полужирный или обычный по единому стилю.
- Текст сносок: Times New Roman, 8 pt.
- Номер сноски в тексте: Times New Roman, 8 pt, надстрочный индекс.
- Колонтитулы: Times New Roman, не крупнее основного текста.

Правила:

1. Основной размер шрифта не уменьшать ниже 12 pt.
2. Табличный шрифт можно уменьшать до 11 pt только для сохранения читаемости и попадания таблицы в печатную область.
3. Шрифт ниже 11 pt в таблицах использовать нельзя без отдельного ручного решения.
4. Все русские символы, знаки градуса, греческие буквы, индексы и специальные символы должны отображаться без квадратов и подмен.
5. Шрифты должны быть встроены в PDF или должны гарантированно отображаться при открытии файла.

## 6. Абзацы и межстрочные интервалы

1. Основной текст выравнивать по ширине либо по левому краю; выбранный режим применять единообразно.
2. Межстрочный интервал должен обеспечивать читаемость при 12 pt: ориентир 14-14.5 pt.
3. Не допускать чрезмерно разреженного текста.
4. Случайные большие межабзацные пустоты запрещены.
5. Первый абзац после заголовка должен быть связан с заголовком правилом keep-with-next.
6. Абзац не должен распадаться так, чтобы на странице оставалась одна последняя или одна первая строка, если это можно избежать без ухудшения вёрстки.
7. Не вставлять ручные пустые строки для имитации переносов.
8. Неразрывные пробелы применять в технических связках, где разрыв недопустим: `рисунок N`, `таблица N`, число + единица, инициалы/сокращения + номер.

## 7. Заголовки

1. Все номера разделов и подразделов сохранять точно.
2. Заголовок не оставлять внизу страницы без последующего текста.
3. Если первый абзац после заголовка не помещается, заголовок переносится вместе с ним.
4. Если в нижней четверти страницы идут два подряд нумерованных заголовка без достаточного связанного текста, оба переносятся на следующую страницу единым блоком.
5. Нельзя переносить заголовок механически, если на текущей странице помещается минимальный связанный блок.
6. Нельзя переносить таблицу или Figure так, чтобы объект оказался в другом разделе-владельце.

## 8. Списки

1. Сохранять тип и уровень исходного списка:
   - нумерованный список `1)`, `2)`, `3)`;
   - буквенный список `a)`, `b)`, `c)`;
   - маркированный список с тире/длинным тире;
   - вложенные уровни.
2. Не преобразовывать списки в обычные абзацы.
3. Не объединять разные пункты списка в один абзац.
4. Не разрывать маркер и первую строку пункта между страницами.
5. Для вложенных списков использовать висячий отступ, чтобы текст пунктов начинался на одном уровне.
6. Между пунктами списка использовать минимальный интервал; случайные большие пустоты запрещены.
7. Если пункт списка содержит связанный подпункт, примечание или формулу, сохранять их вместе насколько возможно.
8. Ссылки внутри пунктов списка оформлять так же, как в основном тексте.

## 9. Переносы текста и технических выражений

1. Автоматические переносы должны быть безопасными для русского технического текста.
2. Нельзя разрывать:
   - число и единицу измерения;
   - знак `±` и значение;
   - `рисунок N`, `таблица N`, `раздел N`;
   - обозначения стандартов, например `API 551`, `ASME B31.3`, `IEC 60529`, `ISO 5167-2`;
   - диапазоны температур и давлений;
   - химические формулы и марки материалов;
   - номера пунктов и заголовков.
3. В единицах измерения и диапазонах сохранять технический смысл: `°C`, `°F`, `%`, `MPa`, `kPa`, `psi`, `in.`, `mm` и т.п.
4. Длинные внешние обозначения стандартов допускается переносить только по безопасным разделителям: пробел, дефис, косая черта, запятая.
5. Висячие одиночные строки в начале/конце страницы устранять средствами вёрстки, а не удалением текста.

## 10. Таблицы

### 10.1. Общие правила

1. Обычные таблицы документа реконструировать как таблицы, а не вставлять картинкой.
2. Название таблицы размещать над таблицей.
3. Название таблицы и первая строка таблицы не должны разделяться переносом страницы.
4. Таблица не должна выходить за правую границу печатной области.
5. Текст в ячейках должен переноситься и оставаться читаемым.
6. Структуру строк, столбцов и объединённых ячеек сохранять.
7. Числовые значения, размерности, диапазоны и технические обозначения не изменять без необходимости.
8. Таблица должна идти после первого логического упоминания, если помещается.

### 10.2. Перенос таблиц

1. Если таблица целиком помещается на пустой печатной странице, её нельзя разрывать.
2. Если таблица не помещается в остаток страницы, весь блок таблицы переносится на следующую страницу.
3. Перед перенесённой таблицей пустое место можно заполнить только допустимым последующим текстом по правилам стоп-элементов.
4. Если таблица физически не помещается на одной странице по высоте, её можно переносить построчно с повтором шапки.
5. При переносе большой таблицы на следующую страницу добавлять пометку вида `Продолжение таблицы N`.
6. Нельзя переносить строку таблицы так, чтобы её текст стал нечитаемым или потерял связь с ячейками.

### 10.3. Ширина таблиц

1. Сначала применять перенос текста внутри ячеек.
2. Затем допускается уменьшение шрифта таблицы до 11 pt.
3. Затем допускается умеренное уменьшение внутренних отступов ячеек.
4. Нельзя уменьшать таблицу так, чтобы текст стал визуально меньше 11 pt.
5. Если таблица всё равно не помещается, объект отправляется на ручную проверку, а не обрезается.

## 11. Рисунки / Figure

### 11.1. Stage 1

1. Figure вставляется как оригинальное crop-изображение из утверждённого реестра.
2. Самодельная переобрезка запрещена, если есть approved crop.
3. Внутренние надписи Figure на Stage 1 не заменяются.
4. Под Figure допускается русская подпись текстом.
5. Рисунок не должен выходить за печатную область.
6. Рисунок не должен разрываться переносом страницы.
7. Подпись рисунка размещается под рисунком.
8. Подпись и рисунок не должны разделяться переносом страницы.

### 11.2. Финальная информационная полнота Figure

1. В финальной сборке использовать approved-рисунки с переведёнными надписями, если такой approved-пакет принят.
2. Если финальная версия включает Stage 2-расшифровку Figure, все содержательные внутренние надписи, NOTE, встроенные таблицы и пояснения внутри Figure должны иметь русскую передачу.
3. Оригинальный чертёж без перевода может использоваться только на рабочих стадиях или как ручное исключение, если это явно утверждено.
4. Встроенные таблицы внутри Figure реконструировать как таблицы, если они являются содержательными.
5. Встроенные NOTE внутри Figure оформлять как отдельные примечания/таблицу примечаний.
6. Чистые числовые размеры и условные символы не нужно превращать в переводной список, но они должны оставаться видимыми в оригинальном/approved рисунке и трассируемыми в JSON.
7. Нельзя удалять Figure-абзацы под предлогом того, что они «не входят в crop»; такой текст должен быть помещён в основной поток или в утверждённую расшифровку.

## 12. Размещение таблиц и рисунков

1. Таблица размещается сразу после абзаца первого упоминания, если помещается.
2. Рисунок размещается после первого логического упоминания с учётом заполнения страницы.
3. Если объект не помещается после первого упоминания, он переносится на следующую страницу.
4. При переносе объекта заполняется только пустота текущей страницы первого упоминания.
5. Заполнение пустоты продолжается до первого стоп-элемента или до исчерпания места.
6. Перенесённый объект идёт на следующей странице перед дальнейшим текстом, который не был использован как заполнитель.
7. Несколько объектов, упомянутых в одном абзаце, считаются связанной группой.
8. Объекты группы идут подряд в порядке упоминания.
9. Если вся группа не помещается, размещается помещающаяся часть группы по порядку, без перестановки объектов.

## 13. Стоп-элементы при заполнении пустот

Заполнение пустого места перед перенесённой таблицей или рисунком должно остановиться перед:

1. новым нумерованным заголовком;
2. первым упоминанием нового объекта;
3. другой таблицей или рисунком;
4. подписью другого объекта;
5. NOTE/примечанием, явно относящимся к перенесённому объекту;
6. сноской, если её ссылка осталась на предыдущей странице;
7. началом нового раздела-владельца.

## 14. Раздел-владелец объекта

1. Раздел-владелец таблицы/рисунка определяется по самому глубокому активному нумерованному заголовку перед первым упоминанием объекта.
2. Объект не должен переноситься через следующий нумерованный заголовок того же или более высокого уровня.
3. Объект не должен «уезжать» в другой подраздел.
4. Если из-за размера объекта это правило невозможно выполнить автоматически, объект отправляется на ручную проверку.

## 15. Примечания NOTE

1. Полное примечание переводить как `ПРИМЕЧАНИЕ` или `ПРИМЕЧАНИЕ N` по контексту источника.
2. Ссылку на примечание в тексте сокращать как `Прим. N`, если это ссылка, а не само примечание.
3. NOTE основного текста и NOTE внутри Figure считаются разными объектами.
4. Группы `NOTE 1`, `NOTE 2`, `NOTE 3` не объединять в один абзац.
5. Примечание, относящееся к таблице или рисунку, не отделять от своего объекта, если это технически возможно.
6. Нельзя исключать NOTE из финального PDF как «внутренний анализ»; исключается только служебный анализ, а исходный NOTE должен остаться как изображение, перевод, примечание или расшифровка.

## 16. Сноски

1. Номер сноски в основном тексте — 8 pt, надстрочный индекс.
2. Текст сноски — Times New Roman, 8 pt.
3. Сноска размещается на той странице, где стоит ссылка.
4. Под сноски заранее резервируется место над нижним колонтитулом.
5. Сноска не должна перекрывать нижний колонтитул.
6. Если сноска не помещается, переносится связанный абзац со ссылкой, а не обрезается сноска.
7. Нумерация сносок должна быть видимой, непротиворечивой и соответствовать ссылкам в тексте.

## 17. Формулы, размеры и единицы

1. Формулы сохранять в читаемом виде.
2. Индексы, степени, греческие буквы и математические знаки не упрощать до потери смысла.
3. Диапазоны значений сохранять полностью.
4. Число и единицу измерения связывать неразрывным пробелом.
5. Температуры оформлять без разрыва между числом и `°C` / `°F`.
6. Не заменять автоматически дюймовые обозначения на русский текст внутри числовых размеров, если это может исказить источник.
7. В заголовках таблиц единицы можно переводить, если это не меняет техническую запись.

## 18. Ссылки и PDF-аннотации

1. Все ссылки на рисунки, таблицы, разделы, примечания и сноски должны быть активными PDF-аннотациями, если цель находится в финальном PDF.
2. Внутренние ссылки должны вести на правильные якоря текущего PDF.
3. В финальной сборке не должно оставаться временных ссылок на исходный PDF для объектов, уже включённых в финальный PDF.
4. В REVIEW-пакетах допустимы временные внешние target-ссылки на исходную PDF-страницу, если объект ещё не включён в текущий пакет.
5. Упоминания `рисунок N` и `таблица N` оформлять единообразно: тёмно-синий цвет, полужирное начертание, рабочая ссылка.
6. Между словом `рисунок`/`таблица` и номером ставить неразрывный пробел.
7. Не должно быть цветного текста без ссылки, если по правилу он должен быть ссылкой.
8. Ссылки на внешние стандарты можно оставлять как обычный текст, если исходный документ не содержит активной внешней ссылки; главное — не искажать обозначение стандарта.
9. После сборки PDF обязательно проверить наличие и координаты link annotations.

## 19. Содержание, перечни рисунков и таблиц

1. Если финальная версия включает содержание, оно должно быть переведено и синхронизировано с фактическими страницами русского PDF.
2. Содержание должно ссылаться на страницы русского PDF, а не на исходные API-страницы.
3. Перечни рисунков и таблиц должны иметь рабочие ссылки на объекты в русском PDF.
4. При изменении пагинации содержание и перечни должны пересобираться.

## 20. Служебное содержимое, запрещённое в финальном PDF

В финальный PDF не включать:

- левую OCR/английскую колонку из REVIEW HTML;
- raw OCR;
- служебные OCR-блоки;
- JSON;
- HTML/CSS/JS-код;
- кнопки, фильтры, элементы поиска, навигацию HTML-интерфейса;
- отчёты проверки;
- debug/TODO/REBUILD/служебные маркеры;
- внутренний анализ Figure как служебный JSON/отчёт.

Это правило не разрешает удалять исходное содержательное тело документа. Исходный текст, таблицы, NOTE и содержательные элементы Figure должны быть представлены в финальном PDF.

## 21. Проверка после генерации PDF

После каждой генерации PDF выполнить:

1. открытие PDF без ошибок;
2. рендер всех страниц в изображения;
3. визуальную проверку контрольных страниц;
4. проверку проблемных страниц вокруг перенесённых объектов;
5. проверку, что текст, таблицы, рисунки, подписи и сноски не перекрываются;
6. проверку, что нет обрезанного текста и объектов за правой границей;
7. проверку link annotations;
8. проверку шрифтов/глифов;
9. проверку отсутствия служебного содержимого;
10. проверку, что количество страниц зафиксировано в кратком отчёте.

## 22. Критические условия браковки PDF

PDF считается непригодным к выдаче, если обнаружено хотя бы одно из следующих нарушений:

1. потерян исходный содержательный абзац, NOTE, таблица, сноска или Figure;
2. текст или таблица обрезаны;
3. объект выходит за печатную область;
4. подпись отделена от рисунка/таблицы;
5. сноска не на странице ссылки или перекрывает колонтитул;
6. внутренние ссылки массово не работают;
7. есть цветные «ссылочные» упоминания без аннотаций;
8. есть raw OCR, JSON, HTML-код или debug-маркеры;
9. неверные поля или базовый шрифт;
10. рисунок переобрезан не по approved crop без ручного решения;
11. таблица вставлена картинкой вместо реконструкции, если она является обычной таблицей документа;
12. Figure/таблица перенесены в чужой раздел.

## 23. Правило выдачи пользователю

При выдаче файлов в чат передавать только:

1. ссылку на ZIP/файл;
2. краткую сводку, что сделано;
3. краткий отчёт проверки.

Не выводить в чат HTML, JSON, таблицы, большие фрагменты правил, внутренние отчёты и содержимое файлов.

- [ ] Таблица размещена сразу после абзаца первого упоминания, если помещалась.
- [ ] Рисунок размещён после первого логического упоминания.
- [ ] Если объект перенесён, пустота заполнена только допустимым последующим текстом.
- [ ] Заполнение пустоты остановлено перед стоп-элементом.
- [ ] Перенесённый объект идёт перед дальнейшим текстом, не использованным как заполнитель.
- [ ] Группа объектов сохранена в порядке упоминания.
- [ ] Объект не перенесён в чужой раздел.
- [ ] Раздел-владелец определён по самому глубокому активному нумерованному заголовку.

## 10. NOTE и сноски

- [ ] NOTE переведены как `ПРИМЕЧАНИЕ` или `ПРИМЕЧАНИЕ N` по контексту.
- [ ] Ссылки на NOTE сокращены как `Прим. N`, если это ссылка.
- [ ] NOTE основного текста и NOTE внутри Figure не смешаны.
- [ ] Группы `NOTE 1`, `NOTE 2`, `NOTE 3` не объединены в один абзац.
- [ ] NOTE, относящиеся к объекту, не отделены от него без необходимости.
- [ ] Сноска размещена на странице ссылки.
- [ ] Сноска не перекрывает нижний колонтитул.
- [ ] Если сноска не помещалась, перенесён связанный абзац со ссылкой.
- [ ] Нумерация сносок видимая и соответствует ссылкам.

## 11. Формулы, размеры и единицы

- [ ] Формулы сохранены читаемо.
- [ ] Индексы, степени, греческие буквы и математические знаки не упрощены до потери смысла.
- [ ] Диапазоны значений сохранены полностью.
- [ ] Число и единица связаны неразрывным пробелом.
- [ ] Температуры не разорваны между числом и `°C` / `°F`.
- [ ] Дюймовые обозначения не заменены автоматически внутри числовых размеров, если это искажает источник.
- [ ] В заголовках таблиц единицы переведены только если это не меняет техническую запись.

## 12. Ссылки и аннотации

- [ ] Все внутренние ссылки на рисунки, таблицы, разделы, примечания и сноски являются активными PDF-аннотациями, если цель есть в PDF.
- [ ] Внутренние ссылки ведут на правильные якоря текущего PDF.
- [ ] Нет временных ссылок на исходный PDF для объектов, уже включённых в финальный PDF.
- [ ] Упоминания `рисунок N` и `таблица N` оформлены единообразно.
- [ ] Между словом `рисунок`/`таблица` и номером стоит неразрывный пробел.
- [ ] Нет цветного текста без ссылки, если он должен быть ссылкой.
- [ ] Проверены координаты link annotations.

## 13. Содержание и перечни

- [ ] Содержание, если включено, переведено.
- [ ] Содержание синхронизировано с фактическими страницами русского PDF.
- [ ] Содержание ссылается на страницы русского PDF, а не исходного API.
- [ ] Перечни рисунков и таблиц имеют рабочие ссылки на объекты.
- [ ] При изменении пагинации содержание и перечни пересобраны.

## 14. Запрещённое служебное содержимое

- [ ] Нет левой OCR/английской колонки.
- [ ] Нет raw OCR.
- [ ] Нет служебных OCR-блоков.
- [ ] Нет JSON.
- [ ] Нет HTML/CSS/JS-кода.
- [ ] Нет кнопок, фильтров, элементов поиска, навигации HTML-интерфейса.
- [ ] Нет отчётов проверки внутри PDF.
- [ ] Нет debug/TODO/REBUILD/служебных маркеров.
- [ ] Нет внутреннего анализа Figure как служебного JSON/отчёта.

## 15. Проверка после генерации

- [ ] PDF открывается без ошибок.
- [ ] Все страницы отрендерены в изображения для проверки.
- [ ] Контрольные страницы визуально проверены.
- [ ] Проблемные страницы вокруг перенесённых объектов проверены.
- [ ] Текст, таблицы, рисунки, подписи и сноски не перекрываются.
- [ ] Нет обрезанного текста.
- [ ] Нет объектов за правой границей.
- [ ] Проверены link annotations.
- [ ] Проверены шрифты/глифы.
- [ ] Проверено отсутствие служебного содержимого.
- [ ] Количество страниц зафиксировано в кратком отчёте.

## 16. Критические условия браковки

- [ ] Нет потерянного содержательного абзаца, NOTE, таблицы, сноски или Figure.
- [ ] Нет обрезанного текста или таблиц.
- [ ] Нет объектов за печатной областью.
- [ ] Нет отделённой подписи от рисунка/таблицы.
- [ ] Нет сноски не на странице ссылки.
- [ ] Нет сноски, перекрывающей колонтитул.
- [ ] Нет массово неработающих внутренних ссылок.
- [ ] Нет цветных «ссылочных» упоминаний без аннотаций.
- [ ] Нет raw OCR, JSON, HTML-кода или debug-маркеров.
- [ ] Поля и базовый шрифт корректны.
- [ ] Нет переобрезанного не по approved crop рисунка без ручного решения.
- [ ] Обычная таблица документа не вставлена картинкой вместо реконструкции.
- [ ] Figure/таблица не перенесены в чужой раздел.
